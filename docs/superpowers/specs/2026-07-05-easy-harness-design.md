# 이지 하네스 (Easy Harness) — 설계 문서

- 작성일: 2026-07-05
- 상태: 사용자 승인 대기 (브레인스토밍 설계 승인 완료, 문서 리뷰 단계)
- 관련: [Threads 발표 게시물](https://www.threads.com/@needsbuilder/post/DaZWRbJASRy)

## 1. 개요

이지 하네스는 AI 입문자가 클릭 몇 번으로 AI 코딩 하네스(Claude Code 등 AI 모델을 감싸 실제 작업을 시켜주는 실행 도구)를 **설치 → 인증 → 첫 실행 확인 → 관리(업데이트/삭제)** 까지 끝내게 해주는 맥/윈도우 데스크톱 앱이다.

기존 EasyClaw(OpenClaw 전용 인스톨러)의 후속 기획이지만, **코드·UI/UX·로고·아이콘·마스코트를 일절 참고하지 않고 완전히 새로 만든다.** 특히 기존 주황색 집게 마스코트 콘셉트는 디자인 탐색에서 명시적으로 배제한다.

### 목표

- AI 입문자가 터미널 지식 없이 하네스를 "바로 쓸 수 있는 상태"로 만든다.
- 새 하네스/도구 추가가 레시피 파일 하나 추가로 끝나는 확장 구조.
- 맥·윈도우 모두 보안 경고창 없이 설치되는 신뢰 가능한 배포.

### 비목표 (v1에서 하지 않는 것)

- 리눅스 지원 (v2 이후 검토)
- 영어 등 다국어 UI (v1은 한국어 전용)
- 하네스 사용법 교육 콘텐츠 (설치·인증·첫 실행 확인까지만 책임)
- 기존 EasyClaw와의 마이그레이션/호환

## 3. v1 라인업 (10종)

**하네스 6종**: Claude Code, Codex/lazycodex, GajaeCode, OpenClaw, Hermes, OpenCode

**오픈소스 4종**: insane-search, k-skill, korean-law-mcp, im-not-ai

각 도구의 정확한 설치·인증 절차는 변동이 잦으므로, 레시피 작성 시점에 공식 문서/저장소를 라이브로 확인해 반영한다(9절 "라이브 검증 목록").

## 4. 사용자 여정 (화면 6단계)

1. **환영 + 환경 진단** — 첫 실행 시 OS·아키텍처·Node.js·git 등을 자동 점검, "준비 OK / 준비물 N개 필요" 를 쉬운 말로 표시. 진단만 하고 설치는 하지 않는다(설치는 마법사에서 필요할 때 자동).
2. **카탈로그** — 10종 도구를 카드로 나열. 카드마다 한 줄 쉬운 설명, 무료/요금제 필요 표시, 입문자 추천 배지. 설치됨 여부도 카드에 표시.
3. **설치 마법사** — 선택한 도구의 레시피를 실행. 사전 준비물 자동 해결 → 본체 설치. 진행 표시는 "지금 Node.js를 설치하고 있어요 (2/4단계)" 식의 쉬운 말 + "자세한 로그 보기" 토글.
4. **인증 도우미** — 설치 직후 이어지는 로그인/키 설정 (6절).
5. **첫 실행 확인** — 레시피의 verify 단계로 실제 동작 테스트, 성공 시 ✅ 배지.
6. **대시보드(관리)** — 설치 목록·버전·업데이트 있음 표시, 업데이트/삭제 버튼. 앱 자체 자동 업데이트 알림도 여기서.

## 5. 아키텍처

```
┌──────────────────────────────────────────┐
│ Frontend: React + TS + Tailwind (WebView)│
│  - 카탈로그·마법사·대시보드 UI           │
│  - 내장 터미널(xterm.js): 대화형 인증    │
│    단계에서만 노출                        │
├──────────────────────────────────────────┤
│ Rust 코어 (Tauri 2)                      │
│  ① 환경 진단기 (env prober)              │
│  ② 레시피 엔진 (recipe engine)           │
│  ③ 스텝 러너 (step runner)               │
│  ④ 상태 저장소 (state store)             │
└──────────────────────────────────────────┘
```

- **환경 진단기**: OS/아키텍처(Apple Silicon·x64)/설치된 도구·버전 감지.
- **레시피 엔진**: 레시피(JSON)를 해석하고 플랫폼에 맞는 단계 목록을 만든다.
- **스텝 러너**: 단계를 순차 실행. 명령 실행, 파일 다운로드, 브라우저 열기, 비밀값 입력 수신, PTY 세션 등 단계 타입별 실행기를 가진다. 로그는 Tauri 이벤트로 실시간 프론트 스트리밍.
- **상태 저장소**: 설치 내역·버전·인증 완료 여부를 로컬 파일에 기록. 대시보드의 데이터 소스.
- **내장 터미널**: xterm.js + PTY(tauri-plugin-pty 계열, 구현 시 최신 상태 확인). 대화형 인증이 필요한 레시피에서만 사용.

### 레시피 시스템

- 레시피 = 도구 1종의 설치 지식을 담은 JSON 파일. 구조(개요):
  - `id`, `name`, `easyDescription`(입문자용 한 줄), `pricing`(무료/요금제 표시)
  - 플랫폼(mac/windows)별 섹션: `detect`, `prerequisites`, `install`, `auth`, `verify`, `update`, `uninstall`, `rollback`
  - 단계 타입: `check_command`, `run_command`, `download_run`, `open_url`, `input_secret`, `pty_session`, `path_check` 등
- **사전 준비물 공용 레시피**: Node.js LTS·git 등은 공용 레시피로 분리, 여러 하네스가 재사용. 설치 방식은 OS 공식 배포본 사용 — Node.js는 공식 .pkg/.msi 다운로드 실행, git은 맥 Xcode Command Line Tools / 윈도우 Git for Windows (구현 시 라이브 검증).
- **원격 갱신**: 레시피는 앱에 기본 번들 + 실행 시 GitHub 레시피 저장소에서 최신본 수신. 설치법이 바뀌어도 앱 업데이트 없이 레시피만 고쳐 전 사용자에 반영. 레시피 번들에는 ed25519 계열 서명을 붙이고 앱이 공개키로 검증해 위·변조를 차단한다.
- **확장**: 새 도구 추가 = 레시피 파일 1개 추가 + 카탈로그 메타데이터. 코드 변경 불필요가 원칙.

## 6. 인증 도우미 (3패턴)

1. **브라우저 로그인형** (예: Claude Code, Codex): "로그인 시작" 버튼 → 앱이 해당 CLI의 로그인 플로우를 실행, 브라우저가 열리면 단계별 그림 안내 → 완료 자동 감지.
2. **API 키 입력형**: 발급 페이지 바로가기 + 마스킹된 입력 폼 → 형식 검증 + 실동작 테스트 후 해당 도구의 자체 설정 파일에 기록. **앱은 키를 보관하지 않는다** — 도구에 전달만 하고, 화면·로그·진단 파일에 노출 금지.
3. **대화형 터미널형**: 내장 터미널을 띄우고 옆에 "지금 이걸 물어보는 거예요 → 이렇게 답하세요" 한국어 도움말 패널 병행 표시.

각 레시피의 `auth` 섹션이 자기 패턴과 안내 문구·이미지를 선언한다.

## 7. 에러 처리

- 실패 단계마다: 쉬운 말 요약 + 해결 제안 + 재시도 버튼.
- 흔한 실패(네트워크 단절, 권한 부족, PATH 미반영, 보안 소프트웨어 차단)는 stderr/종료코드 패턴 매칭으로 감지해 맞춤 안내.
- **진단 파일 내보내기**: 로그 묶음을 zip으로 저장(비밀값 자동 마스킹) → 커뮤니티 문의에 첨부.
- **롤백**: 설치 중단·실패 시 레시피의 rollback 단계로 반설치 상태를 정리.

## 8. 테스트 전략

- Rust 유닛 테스트: 레시피 파서, 스텝 러너(명령 실행은 mock), 상태 저장소.
- **드라이런 모드**: 실제 설치 없이 레시피 단계 해석·순서를 시뮬레이션. 10종 × 2 OS 조합을 CI에서 자동 검증.
- 프론트: 핵심 컴포넌트(마법사 진행, 에러 표시) 단위 테스트.
- 출시 전 스모크 테스트: 깨끗한 맥/윈도우 VM에서 10종 실제 설치·인증·verify 통과 확인 (수동 체크리스트).

## 9. 라이브 검증 목록 (구현 시점에 웹/공식 문서로 재확인)

시간이 지나면 바뀌는 정보라 스펙에 박제하지 않고, 구현 각 단계 시작 시 라이브 확인한다:

1. 하네스 6종 + 오픈소스 4종 각각의 최신 설치 명령·지원 OS·인증 방식 (특히 윈도우 네이티브 지원 여부 vs WSL 필요 여부)
2. Apple Developer Program 가입 절차·연회비, 윈도우 코드 서명 옵션(Azure Trusted Signing 등)의 최신 가격·한국 개인/사업자 가입 가능 여부
3. tauri-plugin-pty 등 PTY 플러그인의 최신 유지보수 상태 (2026-07 확인 기준 활성)
4. Node.js LTS 현재 버전과 공식 배포본 URL 패턴

## 10. 디자인 워크플로우 (Claude Design + 힉스필드 MCP)

1. **브랜드 시안 2방향**: 힉스필드로 (A) 새 마스코트 캐릭터 방향, (B) 미니멀·모던 방향의 로고·앱 아이콘 시안 각각 복수 생성. 프롬프트에 기존 EasyClaw 주황 집게 캐릭터 배제를 명시.
2. **비교 선택**: 두 방향 시안을 한눈에 비교 가능한 형태로 제시, 사용자가 확정.
3. **디자인 시스템 확정**: 선택 방향으로 Claude Design 프로젝트에서 색·타이포·컴포넌트(카드/버튼/진행바)와 6개 화면 목업 제작.
4. **산출물**: 앱 아이콘(.icns/.ico), 로고, 화면 목업, 디자인 토큰 → Tauri 앱 구현에 그대로 반영.

디자인 확정 후 앱 UI 구현에 착수한다 (재작업 방지).

## 11. 배포·서명·업데이트

- **맥**: Apple Developer Program → Developer ID 서명 + 공증 → .dmg 배포.
- **윈도우**: 코드 서명 → NSIS .exe 또는 .msi 배포 (Tauri 기본 지원 포맷 중 구현 시 선택).
- **자동 업데이트**: Tauri 내장 업데이터. 릴리스 채널은 GitHub Releases.
- **CI**: GitHub Actions에서 맥/윈도우 빌드 + 서명 + 릴리스 자동화.

## 12. 구현 순서 (마일스톤)

1. 브랜드·UI 디자인 (힉스필드 + Claude Design, 10절)
2. 앱 뼈대 (Tauri 2 프로젝트, 화면 라우팅, 상태 저장소) + 레시피 엔진·스텝 러너 + 드라이런
3. 하네스 6종 레시피 (설치→인증→verify, 라이브 검증 포함)
4. 오픈소스 4종 레시피
5. 서명·배포 파이프라인 (계정 가입 포함) + 자동 업데이트
6. 맥/윈도우 VM 스모크 테스트 → v1 출시

각 마일스톤은 별도 구현 계획(writing-plans)으로 상세화한다.
